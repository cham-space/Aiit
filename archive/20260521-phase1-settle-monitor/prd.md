## 一、概述与背景（Overview & Background）

公司运营多条外卖/零售平台渠道（美团、饿了么、抖音、京东、银联等），每日产生大量清分资金流水。清分数据落库在 `eid_reconciliation_master` 历史表中，现有系统缺乏对清分到账状态的自动巡检与告警能力，运营团队只能人工核对，存在以下痛点：

- 遗漏/超时清分发现滞后，影响资金安全
- 金额波动无阈值管理，异常数据难以及时察觉
- 通知依赖人工群发，信息传递效率低

**Phase 1 目标**：在不修改任何 `eid_*` 历史数据表的前提下，对各渠道每日清分状态进行自动扫描、状态判定、预警推送、日报汇总、健康度统计，实现资金归集的全链路可观测，并通过 Hook API 对外开放监控数据。

---

## 二、目标与非目标（Goals & Non-Goals）

### 目标

1. **渠道状态自动巡检**：7 个活跃渠道定时扫描（每日 5 次），按优先级算法判定 8 种检查状态
2. **金额变化监控**：计算同比（YoY）、环比（WoW），超阈值自动升级为 AMOUNT_ANOMALY
3. **预警实时推送**：5 类预警类型 + 3 级告警级别，规则匹配 + 冷却期防抖，企业微信 Webhook 通知
4. **全渠道日报**：健康度进度条 + 全局金额汇总 + 各渠道明细，一次推送覆盖所有渠道
5. **Dashboard API**：全渠道状态计数、健康度百分比、整体状态判定、全局金额聚合
6. **Hook API**：Token 鉴权 + 每日调用限频 + 调用日志审计，对外开放渠道状态查询
7. **健康度指标统计**：每日凌晨自动计算前一日 5 维度加权评分，支持趋势分析
8. **运维配置管理**：渠道配置、预警规则、通知组/联系人、运维人员 CRUD API
9. **Nacos 注册中心**：接入 Nacos，启用服务注册与配置管理

### 非目标

- **前端页面**：所有管理界面尚未开发，当前仅提供 REST API，前端后续迭代处理
- **短信通知**：阿里云短信 SDK 已引入但暂不启用，仅企业微信 Webhook 通知
- **ELM_INSURE（淘宝闪购保险）渠道**：该渠道暂停监控，不在扫描范围内
- **eid_* 历史数据修改**：所有历史对账表均为只读，不做任何写操作
- **历史数据迁移**：不对存量历史数据进行回填或迁移

## 三、用户故事（User Stories & Use Cases）

### US-1：运营人员 — 定时自动巡检与异常预警

> 作为运营人员，我希望系统每日自动巡检所有渠道的清分状态，当发现遗漏、异常、超时等情况时，立即通过企业微信推送预警，以便我第一时间介入处理，保障资金安全。

**验收标准**：
- 系统在每日 09:00 / 12:00 / 15:00 / 18:00 / 21:00 自动执行全渠道扫描，误差不超过 ±2 分钟
- 扫描完成后立即触发预警检查，预警从扫描完成到推送延迟 < 5 分钟
- 同渠道同类型预警在 120 分钟内不重复发送（冷却期防抖）
- 预警消息包含：日期、渠道名称、预警类型、清分金额、账单金额、金额一致性、同比/环比变化率
- 单渠道处理失败时不影响其他渠道的扫描和预警（容错隔离）

### US-2：运营主管 — 每日全渠道日报汇总

> 作为运营主管，我希望每天收到一份全渠道日报汇总，一眼看清今天的资金健康度、整体入账情况和各渠道明细，无需逐个查看单个预警。

**验收标准**：
- 日报包含三个区块：①健康度百分比 + ASCII 进度条 ②全局入账/账单总金额 + 平均同比/环比 ③各渠道明细
- 健康度 = (正常渠道数 + 等待中渠道数) × 100 / 总渠道数，整数截断
- 整体状态：有任意 MISSING/OVERDUE/ABNORMAL/INCONSISTENT/AMOUNT_ANOMALY 时为 WARNING，否则 HEALTHY
- 无数据的渠道不参与同比/环比均值计算（避免零值拉低均值）

### US-3：运维系统 — Hook API 集成渠道状态

> 作为第三方运维系统，我希望通过 Token 鉴权的 Hook API 获取当日各渠道清分状态数据，以便集成到公司统一监控面板。

**验收标准**：
- Token 由 `POST /settle/operator/token/generate` 生成，默认有效期 365 天
- 每次调用需在 Header 或参数中携带有效 Token，无效或过期返回 401
- 每个运维人员每日调用上限 200 次，超限返回 429
- 所有调用记录（成功/失败）写入 `settle_hook_call_log` 表用于审计
- 支持查询全部渠道状态和按渠道码查询单个渠道

### US-4：配置管理员 — 渠道与规则配置管理

> 作为配置管理员，我希望通过 API 管理渠道的时间窗口、预警规则、通知组和联系人，当新增渠道或调整监控策略时无需修改代码。

**验收标准**：
- 支持渠道配置的增、改、查（渠道码、预期开始/结束小时、金额阈值、监控组）
- 支持预警规则的增、改、查（规则类型、告警级别、通知组、冷却时间、适用渠道）
- 支持通知组的增、改、查（企业微信 Webhook URL、启用/禁用状态）
- 支持通知联系人的增、改、查（企业微信用户ID）
- 新增渠道只需在 `settle_channel_config` 表插入一行配置即可生效，无需改代码

### US-5：数据分析师 — 历史健康度趋势

> 作为数据分析师，我希望查看各渠道的历史健康度趋势数据，以便分析渠道清分稳定性，发现潜在风险。

**验收标准**：
- 每日凌晨 01:30 自动计算前一日各渠道 5 项健康度指标
- 健康度公式：准时率×25 + 一致率×30 + (1-异常率)×20 + (1-挂起率)×15 + 金额一致率×10
- 最终得分 0-100，保留 2 位小数
- 支持按日期范围查询健康度趋势（默认最近 30 天）
- 支持按渠道码过滤

## 五、成功指标（Success Metrics & KPIs）

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| 预警推送延迟 | 扫描完成后 < 5 分钟 | 监控 `alert_record.record_send_time - snapshot.snapshot_check_time` |

## 六、验收标准 — Acceptance Criteria（综合清单）

基于以上用户故事，以下条件必须全部通过方可视为 Phase 1 完成：

1. [ ] 定时任务在 09:00 / 12:00 / 15:00 / 18:00 / 21:00 准时触发，日志可查
2. [ ] 7 个活跃渠道全部参与扫描，ELM_INSURE 不参与扫描
3. [ ] 每种检查状态（WAITING / IN_PROGRESS / NORMAL / MISSING / OVERDUE / ABNORMAL / INCONSISTENT / AMOUNT_ANOMALY）均有测试用例覆盖状态判定逻辑
4. [ ] MISSING 和 ABNORMAL 预警级别为 CRITICAL；OVERDUE 和 INCONSISTENT 为 WARN；AMOUNT_ANOMALY 为 INFO
5. [ ] 同渠道同类型预警在冷却期内（默认 120 分钟）不会被重复发送
6. [ ] 企业微信 Webhook 推送消息格式符合规范（日期、渠道、类型、金额、同比/环比）
7. [ ] 日报健康度进度条 10 格固定宽度，健康度计算整数截断
8. [ ] 无数据渠道不参与 Dashboard 的 avgYoyChange / avgWowChange 均值计算
9. [ ] Hook API Token 过期或无效返回 401；每日超限返回 429
10. [ ] 每日凌晨 01:30 自动计算前一日健康度指标，结果写入 settle_health_metric
11. [ ] 健康度得分在 0-100 范围，保留 2 位小数
12. [ ] 单渠道扫描异常不阻塞其他渠道（异常日志记录 + 继续处理）
13. [ ] 分组渠道（JDMS_GROUP / BUSINESS_GROUP) 中任一渠道完成清分即整组合格
14. [ ] 同比/环比变化率存储为小数（如 0.125 = +12.5%），基准金额为零时跳过计算
15. [ ] Nacos 服务注册成功，服务在 Nacos 控制台可见
16. [ ] 所有单元测试通过（`mvn test` 零失败）
